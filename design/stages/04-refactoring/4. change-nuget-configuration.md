# Split NuGet Feed and Package Configuration

## Summary
Replace inline `NuGetSources` with feed-level `Environments` configuration and external package-specific JSON files. Each feed is indexed once with all matching package policies, preserving atomic publication and pruning behavior.

## Configuration
- Replace `IndexerOptions.NuGetSources` with:
  - `Environments`: feed definitions containing `Environment`, `Name`, `ServiceIndex`, and `MaxPackages`.
  - `NuGetSourcesPath`: package-file folder, defaulting to `nuget-sources`.
- Resolve relative `NuGetSourcesPath` values against `AppContext.BaseDirectory`.
- Treat the folder as externally provisioned; do not copy it through the project.
- Load top-level `*.json` files once at startup, ordered by filename.
- Environment variables and command-line arguments may override `Environments` and `NuGetSourcesPath`; package-file fields remain file-owned.
- Missing, inaccessible, or malformed folders/files fail startup with the relevant path or filename.
- A feed with no matching package files is skipped without pruning existing data.

Example appsettings:

```json
{
  "DevContextMcp": {
    "NuGetSourcesPath": "nuget-sources",
    "Environments": [
      {
        "Environment": "public",
        "Name": "nuget.org",
        "ServiceIndex": "https://api.nuget.org/v3/index.json",
        "MaxPackages": 100
      }
    ]
  }
}
```

Example `nuget-sources/Formula.SimpleRepo.json`:

```json
{
  "Environment": "public",
  "PackageId": "Formula.SimpleRepo",
  "MaxVersionsPerPackage": 10,
  "IncludePrerelease": false,
  "IncludeUnlisted": false
}
```

## Implementation Changes
- Replace `NuGetSourceOptions` with feed-level `NuGetEnvironmentOptions` and file-level `NuGetPackageOptions`.
- Add a cached package-file loader used by startup validation and `OptionsIndexingConfigurationProvider`.
- Validate environment slugs, unique source names, valid service indexes, positive limits, required package IDs, and case-insensitive package uniqueness within each environment.
- Fail when a package references an undefined environment or when matching package files exceed a feed’s `MaxPackages`.
- Allow multiple feeds to share an environment; apply every package file for that environment to every matching feed.
- Remove prefix discovery and `PackagePrefixes`; every file represents exactly one package.
- Refactor `IndexSourceDefinition` to contain feed identity plus a collection of package-selection definitions with package-specific prerelease, unlisted, and version-limit settings.
- Update NuGet discovery to process each feed once, retrieve metadata using each package’s policy, deduplicate candidates, and publish one complete retained set per feed.
- Skipping a feed with no package files leaves its existing SQLite records unchanged. Removing one file while others remain causes normal publication pruning to remove that package.
- Preserve source IDs, database schema, citations, environment retrieval behavior, MCP contracts, and CLI exit codes.

## Test Plan
- Test executable-relative and absolute folder resolution, deterministic loading, and external-folder behavior.
- Test missing folders, malformed JSON, invalid package files, duplicate packages, unknown environments, and package-count limits.
- Test multiple feeds sharing one environment and packages being indexed from each feed.
- Test package-specific prerelease, unlisted, and maximum-version policies.
- Verify each feed is published once so package files do not prune one another.
- Verify removing one package file prunes that package when others remain, while an environment with no files is skipped.
- Update unit, pipeline, retrieval, and child-process tests to create package folders and pass `NuGetSourcesPath`.
- Update README, specification, architecture, and test-plan configuration examples.
- Run stale `NuGetSources`/`PackagePrefixes` searches, solution build, and the complete test suite.

## Assumptions
- This intentionally removes inline `NuGetSources` compatibility.
- Package filenames are descriptive only; `PackageId` inside the file is authoritative.
- The same package may be configured in different environments but only once within an environment.
- No database migration or Host configuration change is required.
